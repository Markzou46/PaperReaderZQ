# 质量标准与检查方法

## 翻译质量标准

### 一、结构完整性检查

翻译文件必须包含以下结构元素：

| 检查项 | 要求 | 检查方法 |
|--------|------|----------|
| 标题 | 中文标题存在 | 正则匹配 `^# .+` |
| 摘要 | 摘要章节存在 | 正则匹配 `## 摘要` |
| 章节标题 | 主要章节翻译 | 检查 `## 数字` 格式 |
| 公式 | LaTeX格式保留 | 检查 `$$...$$` 或 `$...$` |
| 表格 | Markdown表格格式 | 检查 `\|.*\|` 模式 |

### 二、长度合理性检查

**基本原则**：翻译行数应与原文篇幅成合理比例

```
质量指标 = 翻译行数 / PDF页数

合格标准：
- 质量指标 >= 3.0：优秀
- 质量指标 >= 2.0：合格
- 质量指标 < 2.0：可能缩水，需人工复核
```

**示例**：
- PDF 30页，翻译 150行 → 指标 5.0，优秀
- PDF 30页，翻译 60行 → 指标 2.0，合格
- PDF 30页，翻译 40行 → 指标 1.33，可能缩水

### 三、内容质量检查

#### 3.1 禁止概括性翻译

**检查方法**：对比原文段落数与翻译段落数

```markdown
❌ 错误（概括性翻译）：
本文提出了一种新方法，在多个任务上表现良好。

✅ 正确（详细翻译）：
本文提出了一种新方法，该方法包含三个核心组件：
（1）基于Transformer的编码器，包含12层注意力层；
（2）多任务联合训练策略，使用加权损失函数；
（3）自适应推理机制，可根据输入复杂度调整计算量。
实验表明，该方法在文本分类、命名实体识别和情感分析
三个任务上分别达到了92.3%、88.7%和90.1%的准确率。
```

#### 3.2 检查未翻译内容

**检查方法**：查找大段英文文本

```bash
# 检测连续5个以上英文单词
grep -E '[A-Za-z]+ [A-Za-z]+ [A-Za-z]+ [A-Za-z]+ [A-Za-z]+' translation.md
```

例外情况：
- 专有名词（如模型名称、数据集名称）
- LaTeX 公式
- 代码片段
- 引用标记

#### 3.3 术语一致性检查

**检查方法**：同一术语在全文中的翻译应一致

```markdown
✅ 正确示例：
首次出现：分组查询注意力（Grouped Query Attention, GQA）
后续提及：分组查询注意力...

❌ 错误示例：
首次出现：分组查询注意力（Grouped Query Attention, GQA）
后续提及：GQA机制...（未保持一致）
后续提及：分组注意...（翻译不一致）
```

## 技术报告质量标准

### 一、结构完整性检查

报告必须包含五个核心部分：

| 章节 | 必需内容 | 检查方法 |
|------|----------|----------|
| 一、论文信息 | 标题、作者、机构、链接 | 检查各字段存在 |
| 二、核心贡献 | 一句话总结 + 详细贡献 | 检查"一句话总结"存在 |
| 三、技术方案 | 具体参数和数字 | 检查数字和表格 |
| 四、实验结果 | 性能数据表格 | 检查表格格式 |
| 五、技术脉络 | 演进关系分析 | 检查对比内容 |

### 二、深度要求检查

#### 2.1 具体参数要求

**每个技术报告必须包含以下类型的具体数字**：

```markdown
✅ 必须包含：
- 模型参数：层数、隐藏维度、注意力头数
- 训练配置：batch size、学习率、训练token数
- 实验结果：具体分数、百分比、相对提升

❌ 禁止空洞描述：
"模型使用了较大的参数规模"
"在多个任务上取得了很好的效果"
"训练了大量的数据"
```

#### 2.2 对比分析要求

**必须包含与相关工作的对比**：

```markdown
✅ 正确示例：
| 模型 | 参数量 | 准确率 | 特点 |
|------|--------|--------|------|
| 本方法 | 7B | 92.3% | 多任务联合训练 |
| Baseline A | 7B | 88.1% | 单任务训练 |
| Baseline B | 13B | 90.5% | 更大参数 |

❌ 缺少对比：
"本方法在多个任务上表现良好。"（无具体对比数据）
```

#### 2.3 技术脉络分析要求

**必须分析在技术演进中的位置**：

```markdown
✅ 必须回答：
1. 这项工作继承了哪些已有技术？
2. 相比前作有哪些创新突破？
3. 为后续工作奠定了什么基础？

✅ 示例：
"DeepSeek LLM作为奠基之作，其Multi-step学习率调度器
被后续V2/V3沿用，GQA设计成为MLA的前身思想..."
```

### 三、长度合理性检查

```
质量指标 = 报告行数

合格标准：
- 报告行数 >= 150行：优秀
- 报告行数 >= 100行：合格
- 报告行数 < 100行：可能缩水，需人工复核
```

## 质量评分方法

### 翻译质量评分

```
总分 = 结构分(30%) + 长度分(30%) + 内容分(40%)

结构分：检查必需章节是否存在
长度分：根据翻译行数/PDF页数比例打分
内容分：人工抽检翻译质量
```

### 报告质量评分

```
总分 = 结构分(25%) + 深度分(40%) + 对比分(20%) + 脉络分(15%)

结构分：五个核心部分是否完整
深度分：是否包含具体参数和数字
对比分：是否与相关工作对比
脉络分：技术演进分析是否深入
```

### 质量等级

| 分数范围 | 等级 | 处理方式 |
|----------|------|----------|
| >= 0.9 | 优秀 | 直接通过 |
| 0.7 - 0.9 | 良好 | 通过 |
| 0.5 - 0.7 | 合格 | 提示可能需要复核 |
| < 0.5 | 不合格 | 标记为需重试 |

## 质量检查自动化

### 自动检查脚本逻辑

```python
def check_translation_quality(translation_file, pdf_pages):
    score = 0.0

    # 结构检查 (30%)
    if has_title(translation_file): score += 0.1
    if has_abstract(translation_file): score += 0.1
    if has_sections(translation_file): score += 0.1

    # 长度检查 (30%)
    lines = count_lines(translation_file)
    ratio = lines / pdf_pages
    if ratio >= 3.0: score += 0.3
    elif ratio >= 2.0: score += 0.2
    elif ratio >= 1.5: score += 0.1

    # 内容检查 (40%) - 需要人工复核
    # 这里只做基础检查
    if has_formulas(translation_file): score += 0.1
    if has_tables(translation_file): score += 0.1

    return score
```

### 人工复核要点

当自动检查分数低于 0.7 时，需人工检查：

1. 是否存在大段未翻译内容
2. 是否为概括性翻译
3. 专业术语是否一致
4. 关键技术细节是否遗漏
